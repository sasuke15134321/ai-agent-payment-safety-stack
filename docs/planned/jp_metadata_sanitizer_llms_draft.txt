# JP Metadata Sanitizer

**Do not call this endpoint. It is not implemented or deployed.**
**This API is a specification draft for market testing only.**

## Status
Specification draft only. Not implemented. Not deployed.

## Position
L4 Pre-Payment Guard / L6 Safety Check

## Purpose
Scan payment metadata (x402/A2A/AtoA) for accidental exposure of personal information, contract details, credentials, or internal memos before transmission to external services.

This API checks:
- Personal information exposure (names, emails, phone numbers, IDs)
- Contract information leakage (sensitive negotiation details, confidential terms)
- Credential exposure (API keys, passwords, secrets, tokens)
- Internal memo inclusion (confidential project context, internal decision rationale)
- Other sensitive content that should not reach external services

## Endpoint
```
POST /api/jp/metadata/sanitize
```

## Price (Planned)
0.03 ~ 0.10 USDC / scan

## When to use this API

1. **Before metadata transmission:** Before sending payment metadata via x402, A2A, or AtoA payment flows, scan for sensitive content
2. **Pre-facilitator check:** Before passing metadata to payment facilitators or blockchain networks, verify no sensitive information is included
3. **Payment safety workflow:** As part of the safety check chain before payment execution

Timing in payment flow:
```
JP Counterparty Check → JP Metadata Sanitizer → Payment execution → JP Payment Evidence Guard → agent-memory-api
```

## When NOT to use this API

- Do NOT expect legal advice about what metadata should or should not be sent
- Do NOT expect compliance judgment or regulatory analysis
- Do NOT expect automatic compliance with privacy regulations
- Do NOT expect the API to redact or mask data itself
- Do NOT expect credential storage or recovery
- Do NOT expect individual identification or tracking
- Do NOT expect the API to execute payments or modify metadata
- Do NOT treat this as a payment facilitator or intermediary

## Example Request

```json
{
  "payment_protocol": "x402",
  "metadata_payload": {
    "payer_agent_id": "agent-translation-001",

}
```

## Relationship to Existing APIs

### agent-security-gateway (L4, live)
- Security Gateway checks **prompt injection and instruction safety**
- This API checks **metadata content safety**
- Both operate **before payment execution**
- Call order: security-gateway (instruction check) → metadata-sanitizer (metadata check) → payment

**Important:** Metadata Sanitizer can be implemented as an **extension or derivative** of agent-security-gateway, or as a separate API. Decision to be made in v0.2+.

### JP Counterparty / Invoice Check (L6, planned)
- Counterparty Check verifies **payment recipient identity**
- This API verifies **metadata safety for transmission**
- Both operate **before payment execution**
- Combined flow: Counterparty Check → Metadata Sanitizer → Payment

### JP Payment Evidence Guard (L5, planned)
- Payment Evidence Guard checks **post-payment service delivery**
- This API checks **pre-payment metadata safety**
- Both generate evidence for monthly audit

### agent-memory-api (L8, live)
- Stores metadata sanitization results encrypted
- Includes sanitization status in audit trail
- Enables traceability of what metadata was considered safe

## Constraints

### Does NOT
- Execute payments directly
- Act as a payment facilitator
- Provide legal advice about privacy regulations
- Provide compliance judgment about data handling
- Store or retain the scanned metadata
- Send metadata to external services
- Identify individuals from metadata content
- Automatically redact or mask sensitive fields

### Does ONLY
- Scan and detect sensitive content
- Flag problematic fields
- Route results to human decision
- Classify metadata safety for audit purposes

## Implementation Notes (v0.1)

In v0.1:
- Metadata scanning is **pattern and keyword-based only**
- No machine learning or AI-based anomaly detection
- No PII database integration
- No automatic redaction capability
